# The Agentic Teacher

The teacher is a Gemini 2.5 Pro agent that produces the final lesson critique. It runs as the last stage of the pipeline (`engine/teach_lesson.py:teach_lesson`), uses the multimodal Gemini API to consume audio + score images + video frames, and can call 11 investigation tools mid-conversation to fact-check measurable claims.

This is the second-most-complex part of v2 (after alignment). It exists because mechanical analysis can identify *what* is happening (tempo dropped 30%, intonation 25 cents flat) but cannot tell the user *what to do about it musically*. That requires a teacher.

## The 5 prompt layers

Every teacher invocation sends Gemini the following, in this order:

### Layer 1: Persona system instruction (instrument-aware)

From `engine/instruments.py:system_instruction_for_profile()`. Template-substituted with:

- `instrument` — "violin", "piano", etc.
- `teacher_examples` — "Itzhak Perlman, Rachel Barton Pine, Augustin Hadelich" (per-instrument)
- `voicing_focus` — instrument-specific framing of voicing/balance
- `category_guidance` — what categories of comment to write
- `intonation_or_voicing` — categorical hint
- `measurements_available` — what deterministic data you have to ground claims
- `measurable_claims_rule` — which tool MUST back which kind of claim
- `video_checklist` — per-instrument visual-critique checklist (bow speed, fingering, posture)

The full template is `SYSTEM_INSTRUCTION_TEMPLATE` (~150 lines) and includes:
- "Pitch spelling — important" (use key-appropriate enharmonic spellings)
- "Your discipline" (perception is yours, measurable claims need fact-checking, drop trivia)
- "Video critique — this is half the lesson" (use frames + watch tool, motion vs pose)
- "Dynamics and voicing — comment on these" (push for phrase-level dynamic shape)
- "What good comments look like" (category guidance)
- "Voice" (direct, encouraging, specific, "you" and "your bow")
- "Output format" (the JSON schema with all required fields)

### Layer 2: Recording briefing

```
# Recording briefing
Repertoire: Bach BWV 1001
Movement: Adagio
Instrument: violin
Measures: 1-9
Student notes: (none)
```

### Layer 2b: Prior takes (continuity)

```
# Prior takes of this piece

This is take #2 of this piece. 1 previous take(s) on record:

## Prior take #1 — 2026-05-14 (bars 1–9)
Student's stated focus: "My first take on the first 9 bars of this piece"
Teacher's summary last time: "A good first reading with promising musical instincts..."
Artistic reading last time: "This Adagio is the grand opening statement..."
What was working last time:
  - You have a good foundational command of the notes
  - ...
Areas to develop last time (CHECK IF THESE IMPROVED):
  - [high] Opening Chord Tone and Voicing: ...
  - [high] Trill Control: ...
Last time's prescribed practice (DID THE STUDENT FOLLOW IT?):
  - Practice the opening chord with a focus on G5 projection
  - ...
Last time's 'for next take' note: "Set the camera at 45° from the right..."
Specific comments last time (5 of 7 surfaced; alert/warn only):
  - [ALERT] m1: Opening chord attack is too harsh
  - [WARN] m1: Melody (G5) is buried in the opening chord
  - ...
```

This is generated by `_build_prior_lesson_context` in `apps/api/main.py` and rendered as Markdown by `_read_prior_context` in `engine/teach_lesson.py`. The teacher is instructed (in the system prompt's "Continuity" section) to:

- Open `summary` with explicit reference to the prior take
- Write `progress_notes` as a diff (better / unchanged / new)
- Address whether last week's prescribed practice was followed
- Don't repeat the same artistic_summary; this take is "where to focus AT THIS STAGE"
- Mark improvements as `what_works`, unresolved issues as `areas_to_develop` with escalated drills

**Bug history**: this initially shipped with `_read_past_comments` looking for `player/comments_enriched.json` — but the actual storage key is `lesson/comments_enriched.json`. That made `teacher_comments: []` every time, so the teacher was getting "there was a prior lesson" with no actual content, leading to follow-up takes that read like fresh first lessons. Fixed in the current version.

If there are no prior lessons, this section reads "(no prior lessons — this is the first take of this piece)" and the teacher's continuity directive becomes a no-op.

### Layer 3: Audio (Gemini Files API)

`audio_16k.wav` (16 kHz mono) uploaded via `genai.Client.files.upload`. The 22050 Hz analysis-grade audio (`audio.wav`) is downsampled because Gemini doesn't need the resolution and the upload is faster.

### Layer 4: Score system images (Files API)

`select_score_pages_for_lesson()` returns the relevant page rasters based on `[first_measure, last_measure]` for the played movement. For Bach Adagio bars 1-9 = page 1 only. For Chopin Op 9 No 2 full = pages 2 and 3.

Each page is uploaded as `image/png` with a labeled context line ("score image 1: page-001.png").

### Layer 5: Sample video frames

`_frame_keys()` returns up to `max_video_frames` (default 16) JPEGs from `artifacts/frames/`. Frames are sampled at 10s intervals across the entire video — for a 132s recording that's ~13 frames inline. Each is preceded by a context line ("video frame 3: frame_0003.jpg").

The teacher is told to inspect every frame and call `watch` (see below) for motion-based questions (bow speed, vibrato, pedal change).

### Layer 6: Evidence digest + score note inventory + tool catalog

In one large text block:

- **Evidence digest** (`engine/prompt_evidence.py:build_evidence_digest`): per-bar capsule of intonation/rhythm/voicing measurements with key-aware spelling. Pre-prefixed with the piece's key signature so the teacher uses correct enharmonic spellings.
- **Score note inventory** (`engine/prompt_inventory.py:build_score_note_inventory`): every note in the played measure range with its stable `note_id`, measure, beat_in_bar, names, pitch_midi. The teacher cites these IDs in `references[]`.
- **Tool catalog** (`agent_tools/catalog.py:tool_catalog_text`): per-instrument-filtered list of available investigation tools with their descriptions and JSON arg schemas.

### Layer 7: The task statement

Closing instruction telling the teacher to listen, examine, investigate, and produce one final JSON code block per the schema. Specific asks:
- 5-12 tool calls (efficient investigation, not exhaustive)
- 8-15 final comments (curated, not all 80+ mechanical ones)
- At least 1-2 visual/technique comments when the camera shows useful info
- Honest distinction between observation, inference, and hypothesis

## Multi-turn agentic loop

`agent/gemini.py:generate_with_tools` runs a multi-turn conversation:

1. Send Layer 1-7 as the initial user content.
2. Gemini may respond with `function_call` parts (e.g. `inspect_chord(start_sec=4.5, end_sec=6.0)`).
3. Tool registry handler executes the call (calls into `agent_tools/inspect_chord.py`).
4. Tool result is added to the conversation as a `function_response` part.
5. Loop until Gemini returns a final text response (no more function calls) OR `max_tool_calls=15` is hit.
6. Extract the final JSON code block from the text.

Each call+response pair is recorded in `analysis/teach_tool_calls.json` for the player's audit panel.

## The 11 tools (`agent_tools/`)

| Tool | Purpose | Cost |
|---|---|---|
| `list_frames` | List all extracted video frames | free |
| `get_frames` | Extract additional frames at high fps over a short window | ffmpeg only |
| `inspect_chord` | Spectral peak ranking for chord members | librosa only |
| `inspect_note` | Per-note pitch/loudness/duration measurements | librosa only |
| `inspect_bar` | All notes in a bar with timing/pitch/loudness | librosa only |
| `measure_tempo` | Tempo over a window via onset spacing | librosa only |
| `measure_trill` | Trill rate (Hz) and evenness | librosa only |
| `measure_dynamics` | Per-note loudness envelope across a window | librosa only |
| `measure_vibrato` | Pitch oscillation rate + width (variable-pitch instruments) | librosa only |
| `inspect_voicing` | Chord-balance + top-voice projection (keyboard only) | librosa only |
| `listen` | **PERCEPTUAL** — extract a clip, send to Gemini Flash, ask a question | Gemini Flash call |
| `watch` | **PERCEPTUAL** — extract a short MP4, send to Gemini, ask a motion question | Gemini Flash call |

Per-instrument filtering (`agent_tools/registry.py:default_tool_registry`):
- `inspect_voicing` is keyboard-only
- `measure_vibrato` is filtered out for fixed-pitch instruments

## The `watch` tool (motion video critique)

Added late in development when the user pointed out: "how does it tell bow speed from a single frame?". Single still frames can't show motion.

`agent_tools/watch.py`:

```python
def watch(storage, session, args):
    # 1. Extract a short MP4 clip via ffmpeg
    #    - Re-encode to h.264 (Gemini-compatible)
    #    - No audio (saves bandwidth)
    #    - Downscale to max_height=480px (saves upload time)
    #    - Cap at 10s (WATCH_MAX_CLIP_SEC)
    # 2. Cache under artifacts/watch_clips/
    # 3. Upload to Gemini (inline if <15 MB, else Files API)
    # 4. Send with motion-aware prompt:
    #    "Watch carefully for MOTION cues that a single still frame cannot show:
    #     bow speed/pressure, vibrato rate and width, pedal-change timing,
    #     finger transitions, hand/wrist motion, posture shifts, breath pacing."
    # 5. Return Gemini's answer
```

The teacher learns when to use which video tool from the system instruction:
- Pose / position / posture → existing inline frames or `get_frames` (just need stills)
- Motion / speed / timing-of-gesture → `watch` (need a real video clip)

## The `listen` tool (perceptual audio Q&A)

`agent_tools/listen.py`. Same pattern as `watch` but for audio:

- Extract a clip ≤ 60s (`LISTEN_MAX_CLIP_SEC`)
- Cache under `artifacts/listen_clips/`
- Upload to Gemini Flash with a question
- Return the answer

Used for things the deterministic tools can't measure: tone color, character, articulation quality, sense of phrasing.

## Output schema

The teacher's final JSON (per the schema in the system instruction):

```json
{
  "session": "...",
  "video_path": "...",
  "movement": "Adagio",
  "repertoire": "Bach BWV 1001",
  "played_measures": [1, 9],
  "summary": "2-3 sentence overview",
  "progress_notes": "comparison with prior lessons (empty if none)",
  "enrichment_notes": "what was investigated, what was found, which tools used most",
  "measure_timestamps": [{"measure": 1, "start": 4.9}, ...],
  "lesson": {
    "artistic_summary": "1-2 paragraphs about the piece's musical meaning",
    "what_works": ["specific things the student is doing well"],
    "areas_to_develop": [{"focus": "...", "priority": "high|medium|low", "exercise": "..."}],
    "this_week_practice": ["concrete daily routine items"],
    "next_take": "what to capture differently next time"
  },
  "comments": [
    {
      "id": "g_001",
      "start": 4.83,
      "end": 10.0,
      "category": "musical|voicing|intonation|rhythm|technique",
      "severity": "info|warn|alert",
      "summary": "<= 60 chars",
      "text": "1-3 sentences ending with a try-this",
      "measure": 1,
      "beat": 1.0,
      "evidence_ref": "perception | tool:inspect_chord | etc",
      "provenance": ["perception", "tool:measure_dynamics(...)"],
      "references": [{"measure": 1, "beat": 1.0, "note_name": "G5", "page": 1, "system_index": 1, "note_id": "m1_b1.00_G5+Bb4+D4+G3"}]
    }
  ],
  "dropped": [
    {"id": "c008", "reason": "single-note off-pulse outlier — below threshold"}
  ]
}
```

`comments_enriched.json` includes the same plus a server-added `note_refs` field (resolved from `references[].note_id`) for the player's hover-overlay logic.

## Chat mode

After the initial lesson, the same teacher can answer follow-up questions through `engine/teach_chat.py`. Chat mode reuses the existing agentic infrastructure rather than duplicating it: `load_instrument_profile`, `system_instruction_for_profile`, `default_tool_registry`, `tool_catalog_text`, the evidence digest, score note inventory, score images, compact lesson audio, video frames, and the same structured tool-call loop. The scope is deliberately narrower than continuity mode: the teacher is scoped strictly to the current lesson and piece, not prior lessons or other masterclasses.

The chat prompt starts from the normal instrument-aware system instruction and appends:

```text
## Chat mode

You already gave the student a structured critique of THIS performance (their original lesson takeaway and comments are below). The student is now asking a follow-up question.

- Stay STRICTLY scoped to this lesson and this piece. Do not invent new analyses of unrelated topics.
- You may consult the audio (listen tool), video frames (watch / get_frames tools), or any inspection tool if it helps you give a better answer. Use them sparingly — at most 5 tool calls per response.
- When citing measures, use the same measure numbering convention as the lesson comments.
- If the question is off-topic for music performance, politely redirect.
- Be concise — chat responses should be 1-3 paragraphs unless a longer explanation is genuinely needed.

Original lesson takeaway:
{LESSON_TAKEAWAY_JSON}

Original lesson comments (severity, bar references, summary only):
{COMMENTS_DIGEST}

Conversation so far (most recent last):
{CHAT_HISTORY}
```

Chat has stricter guardrails than the initial lesson: 2 KB messages, 20 user messages per conversation, 50 user messages per user per UTC day, a Gemini Flash topic guard before any Pro call, and at most 5 tool calls per response.

## Cost

Per lesson:
- 1× score-prep call: ~$0.50 (Pro, multimodal score reading) — only if Audiveris fails
- 1× MIDI-find: ~$0.001 (Flash, JSON pick from candidate list)
- 1× teach call: $0.30-1.50 depending on conversation length (Pro, multimodal, 5-15 tool turns)
- 5-15× tool calls inside teach: each ~$0.01-0.05 if it calls Gemini, otherwise free

Total: typically **$0.30-2.00 per lesson**. The teacher dominates.

## Failure modes

1. **Gemini 503 UNAVAILABLE** — capacity throttling on Pro. Wait 10-30 minutes and retry. The pipeline marks `teach_state=failed` but session is still READY with deterministic comments.

2. **Tool errors propagate as JSON to Gemini** — e.g. `{"error": "clip too long"}`. Gemini usually adapts and tries a smaller window.

3. **Final response not valid JSON** — `_extract_json_block` handles markdown fences and partial JSON. If extraction fails entirely, raw response is saved to `llm/raw_teacher_response.json` for debugging.

4. **Tool call quota exhausted** (`max_tool_calls=15`) — teacher is forced to wrap up with what it has. Adjust via `TeachConfig.max_tool_calls`.

## Why dropped comments exist

The mechanical-comments stage produces 80-120 raw flags per lesson (see `pipeline.md:Stage 10`). Most are noise — single-note pitch outliers at slow practice tempo, off-pulse notes by 100ms in a freely-rubato passage. The teacher reads them all in the evidence digest and selects 8-15 that matter musically.

Each unselected mechanical comment lands in `dropped[]` with a one-line reason. The player's UI shows this as a collapsible expander so the user can audit "what did the teacher decide to skip and why?".
